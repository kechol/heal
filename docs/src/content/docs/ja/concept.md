このページでは heal の *なぜ* を説明します。すぐに使い始めたい方は[クイックスタート](/heal/ja/quick-start/)からどうぞ。

## 課題

AI コーディングエージェントは、依頼された次の変更を作るのが得意です。一方でコードベースは裏側でずっと動き続けます。修正や機能追加を重ねるうちに少しずつ複雑度が積み上がり、特定のファイルばかりが繰り返し書き換えられ、重複したコードがじわじわ入り込む。エージェントはそれを見ていません。実際のコードベースでは、「このファイル、最近触りにくくなったな」と人間が気づいた頃には、リグレッションはもう本番に届いています。

## アイデア

> **コードベースの健全性を、エージェントへのトリガーに変える。**

「リンターを走らせるのを忘れないように」と人間が気を張る代わりに、コードベース自身にシグナルを出してもらいます。

- **コミットのたびに**、post-commit フックが全オブザーバーを 1 回走らせ、Critical / High の finding をコミット出力の中に直接表示します。デーモンなしで「次の一手」が常に見えています。
- **オンデマンドで**、`heal status` が同じ findings を TODO キャッシュに書き出し、同梱の `/heal-code-patch` Claude スキルが 1 コミット 1 修正で消化します。

結果として、人間がエージェントを叩き起こすのではなく、コードベースの側からエージェントを呼び出すループになります。

## コードベース相対の Severity

素朴なしきい値（「CCN ≥ 10 は high」）はプロジェクト間ではうまく機能しません。200 行のスクリプトと 200kloc のサービスは別世界で動いています。heal は各メトリクスを **あなたのコードベース自身の分布** に合わせて calibrate します。あなたのコードベースの上位 10% が High、上位 5% が Critical です。再 calibrate は手動（`heal calibrate --force`）です。コードベースを実際に改善するリファクタが、暗黙のうちにゴールポストを動かしてしまっては困るからです。

文献由来の絶対フロアがパーセンタイル分類器を上下から挟むので、一律に悪いコードベースでもワーストケースは Critical のまま、一律にクリーンなコードベースでも「上位 10% は永遠に赤」のループに縛られません。詳しいラダーは[メトリクス](/heal/ja/metrics/)を参照。

## なぜ Hotspot が大事なのか

heal が同梱する 7 つのメトリクスのうち、**最初に見るべきは Hotspot です**。誰も触らない複雑なファイルは技術的負債です。興味深くはあるが、緊急ではない。これに対して、チームが毎日のように編集している複雑なファイルは、次のバグが生まれる場所です。Hotspot は churn × complexity を掛け合わせて、まさにそういうファイルを浮き上がらせます。スコアが高い = よく触られている **かつ** 読みにくい = 歴史的にリグレッションが集中する場所、ということです。

`heal status` の `🔥` フラグがそれを示します。`Critical 🔥` の finding は `Critical` の 2 倍悪いという話ではなく、**直したコストが本当に返ってくる finding** です。`/heal-code-patch` スキルがデフォルトで `🔥` キューを最初に消化するのも同じ理由です。

## デフォルトはリードオンリー、書き込みはスキル経由

`heal` CLI そのものはソースを書き換えません。修復は同梱の Claude スキル 2 つを通じて行いますが、この 2 つは役割を意図的に分けてあります。

- **`/heal-code-review`** は *考える* スキルです。リードオンリー。キャッシュを系として読み、フラグの立ったコードを深く読み、人間がまだ判断すべきアーキテクチャ的な打ち手を提案します。何かを変える前に、キャッシュが何を言っているのかを *理解* したいときに使います。
- **`/heal-code-patch`** は *動かす* スキルです。機械的な修正だけを行います。ドメイン判断を必要としない確立されたリファクタパターンを使って、1 コミット 1 finding でキャッシュを消化していきます。dirty な worktree では起動拒否、push しない、amend しない。次の finding がアーキテクチャ的な判断を必要とする場合は、そこで停止して `/heal-code-review` に渡します。

この分離が、自律性を信頼できる契約になっています。退屈な修正は勝手に進み、面白い判断は手元に残ります。

## heal ではないもの

- **リンターではありません。** リンターは行単位で指摘します。heal は「どのファイルに、どの順で目を向けるべきか」を伝えます。
- **コードレビュアーでもありません。** その役は Claude が担当し、`/heal-code-review` が指揮します。heal はそのプロンプトと TODO リストを整えるだけです。
- **CI ゲートでもありません。** post-commit フックはコミットが入った *あと* に動きます。個別の PR をブロックするのではなく、コードベースの長期的な動きを追うのが目的です。
- **テストの代わりにもなりません。** heal が見つけるのは構造的な複雑さで、コードの正しさは引き続きテストの仕事です。
