このページでは、heal が何を考えて作られているのかを説明します。す
ぐに使い始めたい方は[クイックスタート](/heal/ja/quick-start/)から
どうぞ。

## 課題

Claude Code をはじめとする AI コーディングエージェントは、目の前の
変更を作るのが得意です。一方でコードベースのほうは、コミットを重ね
るたびに少しずつ姿を変えていきます。修正や機能追加を繰り返すうちに
複雑度が積み上がり、特定のファイルばかりが何度も書き換えられ、似た
ようなコードがじわじわ増えていく。こうした長期的な変化はエージェン
トの視野には入っておらず、外からのシグナルがなければ追いかけている
のは人間だけ、という状態になります。

小規模なプロジェクトであれば、それでも問題ありません。しかし、ある
程度の規模になると、「このファイル、最近触りにくくなったな」と気づ
いた頃にはもうリグレッションが本番に届いている、という後手の保守に
なりがちです。

## heal のアイデア

> **コードベースの健全性シグナルを、エージェントへのトリガーに変える。**

CI がテストの実行を監視するのと同じ感覚で、heal はコードベースの状
態を監視します。

- コミットのたびに、コードベースを計測する（複雑度、Churn、
  Duplication、Hotspot など）。
- 小さなスナップショットを `.heal/snapshots/` に書き出す。
- 次に Claude Code のセッションを開いたとき、前回からの変化のうち
  注目に値するものを差し出す。

その結果、「リンターを走らせるのを忘れないようにしないと」と人間が
気を張らなくても、エージェントの側に構造化された通知が届くようにな
ります。たとえばこんな具合です — _「`init.rs` で先ほど触られた関数
が、いまプロジェクトでもっとも CCN の高い関数になっています。リファ
クタリングを検討してください。」_

## ループ

heal は次の 3 ステップで構成されています。

1. **観測（Observe）** — コミットのたびに健全性メトリクスを集める。
2. **通知（Nudge）** — 注目すべき変化を、ちょうどよいタイミングでエージェントに渡す。
3. **修復（Repair）** — 指摘された問題に対して、エージェントが PR を作る。何をマージしてよいかはポリシーで決める。

最初の 2 ステップは、実際にはこんなふうに動きます。

```
コミット時
─────────────────────────────────────────────────

git commit
    │
    ▼
post-commit フック ──► heal hook commit
                           │
                           ├─ オブザーバーを走らせる（LOC, complexity, churn, …）
                           │
                           └─ スナップショットを書き出す ──► .heal/snapshots/

                  （スナップショットは保存され、次のセッションを待つ）


次の Claude Code セッション
─────────────────────────────────────────────────

claude を起動
    │
    ▼
SessionStart フック ──► heal hook session-start
                            │
                            ├─ 直近のスナップショットと差分を読み込む
                            ├─ しきい値とクールダウンを確認
                            │
                            ├─ しきい値を超えていれば ──► Claude にナッジを渡す
                            └─ 超えていなければ      ──► 何もしない
```

**クールダウン**は、同じ通知が毎セッションのたびに流れ続けるのを防
ぐ仕組みです。一度ルールが発火すると、その後 `cooldown_hours`（デ
フォルトは 24 時間）はそのルールが抑制されます。クールダウンが明け
てもまだしきい値を超えていれば、また発火します。

実際のサイクルは、ざっとこんな流れです。

1. コードを書いてコミットする。heal は裏側で静かにスナップショッ
   トを記録する。
2. 続けて Claude Code を開く。
3. heal は新しいスナップショットを前回のものと比較する。
4. ホットスポットのファイルが入れ替わった、複雑度が急に跳ね上がっ
   た、といった変化があれば、Claude のコンテキストの先頭に短い通知
   が表示される。
5. その場で対処してもよし、`heal check` で詳しく見るもよし、後回し
   にするのも自由。

3 つめの **修復（Repair）** は、このループをさらに一歩先に進めるス
テップです。通知を出すだけで終わらせず、指摘した問題に対する PR を
heal が用意する — ただし、どこまで自動でやらせるかは、明示的なポリ
シーで自分でコントロールします。

## デフォルトはリードオンリー

修復ポリシーを自分で有効化しない限り、heal はソースコードには手を
加えません。どのコマンドも、メトリクスを読むか、Claude に解説して
もらうために渡すか、そのどちらかです。heal が書き込むのは次の 3 か
所だけです。

- `.heal/` — heal 専用のデータディレクトリ
- `.git/hooks/post-commit` — 1 行だけのフック行を一度設置する
- `.claude/plugins/heal/` — `heal skills install` で展開する、こち
  らもオプトイン

自動修復を行うには `policy.action = execute` を明示的に指定する必
要があります。それを設定するまでは、すべての変更は heal の通知を見
たうえでの人間の判断によって行われます。

## なぜメトリクスなのか

heal には 6 つのメトリクスが付属しています。

- **LOC** — プロジェクトの言語構成
- **複雑度（CCN + Cognitive）** — 読むのがしんどい関数
- **Churn** — よく書き換えられているファイル
- **Change Coupling** — 一緒に変更されがちなファイル
- **Duplication** — コピペで増えてしまったコード
- **Hotspot** — Churn × 複雑度 の合成。「コードを犯罪現場として見
  る」視点

どれも昔から研究されてきた定番のメトリクスばかりで、AI 専用に作っ
たものは一つもありません。heal の貢献はメトリクス自体ではなく — そ
れらは何十年も前から存在しています — それらをエージェントのループ
にとってのトリガーとして使い、人間をポーリング役から解放するところ
にあります。

各メトリクスの式は[メトリクス](/heal/ja/metrics/)を参照してください。

## なぜフックで動かすのか

エージェントはコードを書くのは上手ですが、自分のまわりの状態を逐一
見てくれるわけではありません。フックを使えば、コードベースのほうか
ら自分でシグナルを出してもらうことができます。

- **git の post-commit フック**は、コミットが入ったその瞬間にスナッ
  プショットを書きます。デーモンも、スケジューラーも、定期ポーリン
  グもいりません。
- **Claude Code の SessionStart フック**は、セッションが開いた瞬間
  に最新のスナップショットを読みます。エージェントがこれから動き出
  す、まさにそのタイミングでシグナルが届きます。

どちらのフックも、呼ぶ先は同じ `heal` バイナリです。常駐するプロセ
スは何もありません。

## heal ではないもの

- **リンターではありません。** リンターは行単位で指摘します。heal
  は「どのファイルに目を向けるべきか」を伝えます。
- **コードレビュアーでもありません。** その役は Claude が担当しま
  す。heal はそのプロンプトを整えるだけです。
- **CI ゲートでもありません。** post-commit フックはコミットが入っ
  たあとに動きます。個別の PR をブロックするのではなく、コードベー
  スの長期的な動きを追うのが目的です。
- **テストの代わりにもなりません。** heal が見つけるのはあくまで構
  造的な複雑さで、コードの正しさは引き続きテストの仕事です。
