import { Tabs, TabItem } from '@astrojs/starlight/components';

このウォークスルーが完了すると、以下が揃います：

- git リポジトリの中で動いている heal
- キャリブレーション済みの `.heal/` ディレクトリと post-commit フック
- 画面に出る最初の Severity 別 TODO リスト
- （任意）`/heal-fix` でキャッシュを消化する Claude

ソースをクローンしてビルドする必要はありません。所要時間は約 5 分。

## 必要環境

- macOS または Linux
- 評価対象の git リポジトリ（自前のもの、あるいは任意の OSS クローン）
- （任意）[Claude Code](https://claude.com/code) CLI — `check-*` スキ
  ルや `/heal-fix` を使う場合

## 1. `heal` をインストール

環境に合うものを選んでください：

<Tabs syncKey="installer">
  <TabItem label="Homebrew">

```sh
brew install kechol/tap/heal-cli
```

macOS / Linux 共通。各リリースに同梱されるプリビルドバイナリ。

  </TabItem>
  <TabItem label="Cargo">

```sh
cargo install heal-cli
```

`PATH` に Rust ツールチェイン (1.85 以降) が入っているとき。crates.io
からビルドを取得します。

  </TabItem>
  <TabItem label="Shell">

```sh
curl --proto '=https' --tlsv1.2 -LsSf \
  https://github.com/kechol/heal/releases/latest/download/heal-cli-installer.sh | sh
```

`~/.cargo/bin` 配下にプリビルドバイナリを置きます。Rust も Homebrew
も無いときに。

  </TabItem>
</Tabs>

インストール確認：

```sh
heal --version
```

コマンドが見つからない場合、`~/.cargo/bin` が `PATH` に入っているか
確認してください。各方式の詳細は
[Installation](/heal/ja/installation/) を参照。

## 2. リポジトリをセットアップ

git リポジトリの中で実行：

```sh
heal init
```

このコマンドは：

- `.heal/` を作成（`config.toml`、`calibration.toml`、`snapshots/`、
  `logs/`、`checks/`）。
- 各オブザーバを 1 度走らせ、メトリクスごとのコードベース固有のパー
  センタイル分布 (`p75 / p90 / p95`) を計算 — それが
  `calibration.toml`。
- `.git/hooks/post-commit` をインストール。
- 初回 `MetricsSnapshot`（Severity tally 込み）を append。

再実行は安全です。`config.toml` も既存の非 heal フックも `--force`
無しでは触りません。

## 3. コードベースの状態を把握

```sh
heal status
```

各メトリクスのサマリと、前回スナップショットからの delta が出ます。
最初のコミット後はこんな感じ：

```
Primary language: Rust  (18421 LOC across 142 files)
  complexity:        max CCN 28  Cog 42  (172 functions across 73 files)
  hotspot:           max score 640.0
  lcom:              5 / 12 classes ≥ min_cluster_count=2
  Δ since last:      max_ccn  +2  (busy_function)
```

ファイル単位の Severity 別 TODO リストが本命のワークフロー：

```sh
heal check
```

サンプル出力：

```
── HEAL check ──────────────────────────────────────
  Calibrated: 2026-04-30 09:00  (12 findings, head a1b2c3d)

🔴 Critical 🔥 (1)
  src/payments/engine.ts  CCN=28
🟠 High (3)
  src/billing/cart.ts     CCN=14  Cognitive=18
  ...

  Goal: 0 Critical, 0 High  (current: Critical=1  High=3)
  Next: `heal check --severity critical` / `claude /heal-fix`
──────────────────────────────────────────────────
```

このビューは findings を Severity 別にグループ化し、hotspot ファイル
を `🔥` で示し、`CheckRecord` を `.heal/checks/latest.json` に書き
出します — `/heal-fix` が消費する TODO リストです。

`heal check` はリードオンリー。リストを絞るには：

```sh
heal check --metric lcom              # LCOM のみ
heal check --severity critical        # Critical のみ
heal check --feature src/payments     # src/payments/ 配下のみ
heal check --hotspot                  # 低 Severity だが hotspot のファイル
```

## 4. キャッシュを消化（任意）

同梱の Claude プラグインを入れた状態で：

```sh
heal skills install
```

…次のように Claude に TODO リストを 1 finding = 1 commit のペースで
消化させられます：

```sh
claude /heal-fix
```

このスキルは Severity 順 (`Critical 🔥` 優先) に次の non-Ok finding
を選び、ファイルを読み、リファクタを計画 (CCN なら Extract
Function、duplication なら Extract Method など)、変更を適用、テス
トランナを実行、Conventional Commits + `Refs: F#<finding_id>`
trailer 付きでコミット、再スキャンで確認します。

スキル側の制約：

- dirty な worktree では起動拒否。
- 1 finding = 1 commit。
- push しない、amend しない。

キャッシュが空になったらループ終了。`git push` / `gh pr create` は
ユーザーが手動で実行します。

## 5. post-commit nudge と暮らす

`heal init` で post-commit フックがインストール済みです。これ以降、
コミットごとに：

- 各オブザーバが 1 回走る。
- `MetricsSnapshot` が `.heal/snapshots/` に append される。
- Critical / High の finding が commit メッセージ直後に stdout に出
  る — `Medium` と `Ok` は黙っています。

同じ問題は修正されるまで毎コミット nudge に出ます。cool-down はあり
ません — それが仕様です。

キャリブレーションがドリフトしたとき (90 日経過、コードベースのファ
イル数が ±20% 変化、30 日連続で zero Critical) は、nudge の冒頭に
「再較正をおすすめ」のヒントが付きます。準備ができたら
`heal calibrate` を実行してください。再較正は決して自動では走りませ
ん。
