heal には 6 つのメトリクスが付属しています。いずれも AI 専用ではな
く、何十年も使われてきた実績のあるコードヘルスメトリクスです。
heal の貢献は、これらをトリガーとして使うこと — コミットごとに収集
し、しきい値を越えた変化を次の Claude セッションに伝えること — に
あります。

このページでは各メトリクスを概観します。設定項目については
[設定](/heal/ja/configuration/)を参照。

## 推奨される導入順

導入を始めるとき:

1. **LOC** は常に有効 — 設定不要。
2. **Cognitive** は有効のままで問題ありません。**CCN** はデフォル
   ト無効です。古典的な Cyclomatic 値も欲しい場合に有効化します。
3. **Churn** は有効です。参照ウィンドウ内に 10 件ほどコミットが溜
   まると意味のある情報になります。
4. それらが妥当に見えるようになったら、**Hotspot** を有効化します。
   単独メトリクスとしてもっとも実用的です。
5. **Duplication** と **Change Coupling** は診断用です。調査時に有
   効化し、それ以外は切っておくのが良いでしょう。

## LOC — Lines of Code

> _「このコードベースは何で構成されているのか？」_

[`tokei`](https://github.com/XAMPPRocky/tokei) を使って、言語ごとの
コード行数、コメント行数、空行をカウントします。LOC は基礎的なメト
リクスです — 他のメトリクスはこれによる言語検出に依存します（複雑
度はパースできる言語でしか動かず、Hotspot はコミットを複雑度で重み
付けします）。

出力例:

```
Primary: Rust
  Rust         18421 code  2891 comments
  TypeScript    4920 code   612 comments
  Markdown      1830 code     0 comments
```

「primary language」は、リテラート言語ではない言語のうちもっともコー
ド行数が多いものです。ドキュメント中心のリポジトリでも実装言語に解
決されるよう、Markdown と Org は意図的に除外されています。

LOC は常に有効でトグルはありません。`tokei` がファイル単位でキャッ
シュするため、コストはほぼ無視できます。

## Complexity — CCN と Cognitive

> _「どの関数が追いにくいのか？」_

tree-sitter のシングルウォークで計算される、関数単位の 2 つのメト
リクスです。

- **CCN**（Cyclomatic Complexity） — McCabe によるブランチ数。
  各 `if`、`for`、`while`、`case`、`&&`、`||`、`?` ごとに 1 増えま
  す。10 を超えるとボーダーライン、20 を超えるとリファクタ候補とい
  うのが一般的な目安です。
- **Cognitive Complexity** — Sonar の可読性メトリクス。**ネストの
  深さ**を罰し（深くなるほど加算量が増える）、連鎖した論理演算子は
  単一の加算にまとめます。主観的な「読みにくい」の判断と相関が高い
  指標です。

両者とも有用です。CCN は古典的なブランチカウント、Cognitive は知覚
コストの近似値です。

**対応言語**: TypeScript と Rust。今後、JavaScript、Python、その他
の言語が追加されます。

## Churn — ファイルがどれだけ変更されているか

> _「何が動いているのか？」_

リポジトリ内の各ファイルについて、直近 `since_days` ウィンドウ（デ
フォルト 90 日）でそのファイルを触ったコミット数と、追加・削除され
た合計行数を報告します。マージコミットの二重カウントを避けるため、
first-parent 履歴を使います。

Churn が高いこと自体は必ずしも問題ではありません — `package.json`
のような定形のファイルは頻繁に変わります。Churn は複雑度と組み合わ
せたときに意味を持ちます（[Hotspot](#hotspot--churn--complexity)を参照）。

**注意点**: リネーム検出は現在無効です。ウィンドウ途中でリネームさ
れたファイルは 2 つのエントリとして現れます。一括フォーマットは
`lines_added`/`lines_deleted` を膨らませます。生の行数が誤解を招き
そうなときは、コミット数を信用してください。

## Change Coupling — 一緒に動くファイル

> _「どのファイルがどのファイルに、暗黙のうちに依存しているか？」_

各コミットで、Change Coupling はそのコミットが触ったファイル集合を
調べます。集合内のすべてのファイルペアについてカウンタが増加します。
十分な数のコミットを経た後、最も高いカウンタは暗黙の依存関係 — イ
ンポートグラフ上では繋がっていないのに、一貫して一緒に変更されるファ
イル — を浮かび上がらせます。

出力例:

```
init.rs  ↔  paths.rs       co-changed 11×
status.rs ↔ snapshot.rs    co-changed  9×
```

これは診断用メトリクスです。高い結合のペアは、2 つのファイルの間に
あるべき抽象が欠けていることを示すことが多いです。

**バルクコミットのキャップ**: 50 ファイル超を触るコミット（一括フォー
マット、依存ライブラリの一括更新など）は、無関係なファイル間に偽の
結合を作らないよう、完全にスキップされます。

## Duplication — コピーされたブロック

> _「どこに重複があるのか？」_

tree-sitter の構文木を歩いて、`min_tokens`（デフォルト 50）サイズ
のトークンウィンドウをマッチさせ、長く続く同一トークン列（Type-1
クローン）を見つけます。整形やホワイトスペースの違いではクローンは
隠れません。一方、変数のリネームは隠れます。

出力例:

```
92 tokens duplicated in:
  - status.rs:4123–5210
  - check.rs:2811–3902
```

長いブロックは抽象が抽出できることを示すサインのことが多いです。短
いブロック（`min_tokens` 付近）は、似た形のボイラープレートに過ぎ
ないこともあります。プロジェクトに合わせて `min_tokens` を調整して
ください。

**対応言語**: 複雑度と同じ（TypeScript、Rust）。

## Hotspot — Churn × Complexity

> _「リグレッションはどこに集中するか？」_

Adam Tornhill が広めた「コードを犯罪現場として見る」視点です。
Hotspot はファイルのコミット数（Churn）に、関数の CCN 合計（複雑度）
を掛け合わせます。

```
score = (weight_complexity × ccn_sum) × (weight_churn × commits)
```

スコアの高いファイルは、頻繁に変更されており、かつ読みにくい — 歴
史的にリグレッションが集中するところです。週次でひとつのメトリク
スだけ確認するなら、これがおすすめです。

重みはどちらもデフォルト `1.0` です。プロファイルが偏ったプロジェ
クトでは調整してください — たとえば、「小さなコミットが多い」だけ
でスコアが支配されないように `weight_complexity` を上げる、など。

式は乗算であるため、複雑度は高いが直近のコミットがないファイルはス
コアが 0 になります。これは意図的です — Hotspot は _アクティブな_
トラブルを特定するためのものであり、過去の負債を掘り返すためのもの
ではありません。

## heal がこれらをどう使うか

コミットごとに、heal は `MetricsSnapshot` を `.heal/snapshots/` に
書き出します。`heal status` がサマリを表示し、`heal check` はデー
タを Claude にフォーカスされたプロンプトと一緒に渡します。スナップ
ショット間でしきい値が越えられると、SessionStart フックが次の
Claude セッションで通知として表面化させます — ルール一覧は
[Claude プラグイン](/heal/ja/claude-plugin/)を参照。

スクリプティング向けに、JSON の正確な形と保存方法は
[アーキテクチャ › スナップショット](/heal/ja/architecture/#snapshots--メトリクスペイロード)
にまとめています。
