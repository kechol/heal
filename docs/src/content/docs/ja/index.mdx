固有の Severity で分類して、次の Claude セッションに TODO リストとして
渡します。

## なぜ heal なのか

AI コーディングエージェントは、依頼された次の変更を生み出すのは得意
です。一方でコードベースは継続的に変化していきます。複雑度が積み重
なり、同じファイルが何度も触られ、重複したコードが少しずつ増えてい
く。エージェントはこうした長期的な変化を観測しません。外部からのシ
グナルがなければ、それを追えるのは人間だけです。

`heal` はそのギャップを埋めます。git の post-commit フックがコミッ
トごとにメトリクスのスナップショットを記録し、Critical / High の
finding を stdout に出力します。`heal check` は同じ findings をキャッ
シュに書き込み、同梱の `/heal-code-patch` Claude スキルが 1 finding = 1
commit のペースで消化します。

## 概要

<CardGrid>
  <Card title="7 つの較正されたメトリクス" icon="seti:rust">
    LOC、複雑度 (CCN + Cognitive)、Churn、Change Coupling (一方向 +
    双方向)、Duplication、Hotspot、LCOM — それぞれが
    コードベース自身の分布に対するパーセンタイルで分類されます。
  </Card>
  <Card title="フック駆動" icon="puzzle">
    git の post-commit フックがスナップショット書き込みと Severity nudge
    の両方を担当。デーモンもスケジューラもなく、オブザーバは コミットごとに 1 回だけ走ります。
  </Card>
  <Card title="CLI はリードオンリー、修復は skill 経由" icon="approve-check">
    CLI 自体はソースを書き換えません。修復は `/heal-code-patch` を通じて 1 finding = 1 commit、dirty
    worktree では起動拒否、push しない、 amend しない、というルールで進みます。
  </Card>
  <Card title="単一バイナリ" icon="rocket">
    `cargo install heal-cli` で CLI が入ります。Claude スキルセットは バイナリに同梱されており、`heal
    skills install` で展開します。
  </Card>
</CardGrid>
