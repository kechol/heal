After completing this walkthrough you will have:

- heal running inside one of your git repositories
- A calibrated `.heal/` directory and the post-commit hook installed
- The first Severity-grouped TODO list on screen
- (Optional) Claude draining the cache via `/heal-code-fix`

No source clone or manual build is required. The walkthrough takes
about five minutes.

## Requirements

- macOS or Linux
- A git repository to evaluate — your own or any open-source clone
- (Optional) the [Claude Code](https://claude.com/code) CLI, for the
  `/heal-code-check` audit and `/heal-code-fix` repair skills

## 1. Install `heal`

Choose whichever method suits your environment:

<Tabs syncKey="installer">
  <TabItem label="Homebrew">

```sh
brew install kechol/tap/heal-cli
```

Available on macOS and Linux. Same prebuilt binary that ships with
each release.

  </TabItem>
  <TabItem label="Cargo">

```sh
cargo install heal-cli
```

Use this when a Rust toolchain (1.85 or newer) is already on the
`PATH`. The build is fetched from crates.io.

  </TabItem>
  <TabItem label="Shell">

```sh
curl --proto '=https' --tlsv1.2 -LsSf \
  https://github.com/kechol/heal/releases/latest/download/heal-cli-installer.sh | sh
```

Installs the prebuilt binary under `~/.cargo/bin`. Useful when
neither Rust nor Homebrew is available.

  </TabItem>
</Tabs>

Verify the install:

```sh
heal --version
```

If the command is not found, ensure `~/.cargo/bin` is on the `PATH`.
See [Installation](/heal/installation/) for details on each method.

## 2. Set up your repository

Inside a git repository, run:

```sh
heal init
```

This command:

- Creates `.heal/` with `config.toml`, `calibration.toml`,
  `snapshots/`, `logs/`, `checks/`.
- Runs every observer once to compute the codebase's percentile
  distribution per metric (`p75 / p90 / p95`) — that's
  `calibration.toml`.
- Installs `.git/hooks/post-commit`.
- Appends the first `MetricsSnapshot`, including the Severity tally.

The command is safe to re-run. The config file and any existing
non-heal hooks are left in place unless `--force` is passed.

## 3. See where the codebase stands

```sh
heal status
```

Prints a summary of every enabled metric and the delta since the
previous snapshot. After the first commit you'll see something like:

```
Primary language: Rust  (18421 LOC across 142 files)
  complexity:        max CCN 28  Cog 42  (172 functions across 73 files)
  hotspot:           max score 640.0
  lcom:              5 / 12 classes ≥ min_cluster_count=2
  Δ since last:      max_ccn  +2  (busy_function)
```

For a per-file Severity-grouped TODO list — the canonical workflow:

```sh
heal check
```

Sample output:

```
── HEAL check ──────────────────────────────────────
  Calibrated: 2026-04-30 09:00  (12 findings, head a1b2c3d)

🔴 Critical 🔥 (1)
  src/payments/engine.ts  CCN=28
🟠 High (3)
  src/billing/cart.ts     CCN=14  Cognitive=18
  ...

  Goal: 0 Critical, 0 High  (current: Critical=1  High=3)
  Next: `heal check --severity critical` / `claude /heal-code-fix`
──────────────────────────────────────────────────
```

This view groups findings under Severity tiers, surfaces hotspot
files with `🔥`, and writes a `CheckRecord` to
`.heal/checks/latest.json` — the TODO list `/heal-code-fix` will consume.

`heal check` is read-only. To narrow the list:

```sh
heal check --metric lcom              # only LCOM findings
heal check --severity critical        # only Critical
heal check --feature src/payments     # only files under src/payments/
heal check --hotspot                  # low-Severity hotspot files
```

## 4. Drain the cache (optional)

With the bundled Claude plugin installed:

```sh
heal skills install
```

…you can audit the cache or drain it through Claude:

```sh
claude /heal-code-check    # architectural reading + prioritised TODO
claude /heal-code-fix      # one finding per commit, in Severity order
```

`/heal-code-check` is read-only — it ingests the cache, deep-reads
the flagged code, and returns an architectural reading plus a
prioritised refactor TODO list (grounded in the metric literature
and module-depth / layered / DDD vocabulary).

`/heal-code-fix` picks the next non-Ok finding in Severity order
(`Critical 🔥` first), reads the file, plans a refactor (Extract
Function for CCN, Extract Method for duplication, etc.), applies the
change, runs your test runner, commits with a Conventional Commits
message + `Refs: F#<finding_id>` trailer, then re-scans to confirm.

Constraints (enforced by the skill):

- Refuses on a dirty worktree.
- One finding = one commit.
- Never pushes; never amends.

When the cache is empty, the loop ends. You then `git push` /
`gh pr create` yourself.

## 5. Live with the post-commit nudge

`heal init` installed a post-commit hook. From this point, every
commit:

- Re-runs every observer once.
- Appends a `MetricsSnapshot` to `.heal/snapshots/`.
- Surfaces every Critical / High finding to stdout right after the
  commit message — `Medium` and `Ok` stay quiet.

The same problem reappears in the nudge every commit until it's
fixed. There is no cool-down — that's the point.

When the calibration drifts (older than 90 days, codebase file count
shifted ±20%, or 30 days of zero Critical), the nudge prepends a
"consider recalibrating" hint. Run `heal calibrate` when ready;
recalibration is never automatic.
